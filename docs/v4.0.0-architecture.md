**Status:** Design locked.
**Decided:** 2026-05-03
**Target ship:** 12-16 weeks from sprint start (quality over speed)
**Branch:** `v4-rewrite` (no public releases until merged)
**v3.19.9 stays live on npm/GitHub the entire time.**

## Mission statement

**Aiden v4.0.0 = Hermes-grade everything, plus Aiden's unique layer.**

Not "Hermes-inspired." Not "ported some patterns." Full feature parity with Hermes Agent's architecture, tooling, skills, memory, security, UX — end to end. Then Aiden's identity stacked on top: honesty enforcement, Pro tier, OAuth subscription onboarding, native Windows, npm distribution.

The result: "Hermes for Windows, with honesty guards and a Pro tier." A real, defensible, marketable product — not a clone, not an inspiration, a fusion.

## Development tooling (the workflow)

Before code starts, set up Graphify on both repos. This is the single highest-leverage workflow improvement for the 12-16 week port.

**What Graphify gives us:**
- Maps Hermes Python codebase (~50k LOC across 100+ files) into a queryable knowledge graph
- Maps Aiden TypeScript codebase as it grows from ~3k to ~11k LOC
- Auto-rebuilds after every git commit (hook installed)
- Claude Code reads the graph before answering — no random grepping
- Native Hermes integration (`graphify install --platform hermes`)

**Use cases during the port:**
1. **Reference lookup:** "Which Hermes file handles tool-calling loop edge cases?" → graph answers in seconds, no need to paste 13.7k LOC
2. **Cross-mapping:** Merge Hermes + Aiden graphs → "What's the Aiden equivalent of `run_agent.py`?" → graph answers
3. **Architecture sanity:** "What depends on `core/aidenAgent.ts`?" → graph shows full impact radius before refactoring

**One-time setup (30 min):**
```powershell
# Install Graphify globally
uv tool install graphifyy

# Map Hermes (reference repo)
cd C:\path\to\hermes-agent
graphify update .
git add graphify-out/ && git commit -m "graphify: map for Aiden v4 reference"

# Map Aiden (active dev repo)
cd C:\Users\shiva\DevOS
graphify update .
graphify hook install   # auto-rebuild on every commit
graphify claude install # Claude Code reads graph before answering
git add graphify-out/ && git commit -m "graphify: setup"
```

> **Note:** Graphify CLI uses `graphify update <path>` to build/refresh a graph (not bare `graphify <path>` as some docs show). Confirmed against graphify 0.6.8.

**`.gitignore` additions for both repos:**
```
graphify-out/manifest.json
graphify-out/cost.json
graphify-out/cache/
```

**No Obsidian.** Graphify's HTML viewer + `GRAPH_REPORT.md` covers the visual navigation use case. Adding Obsidian = two sources of truth, drift, maintenance. Defer indefinitely. If ever needed, `graphify update . --obsidian` generates a vault on demand.

**Privacy:** Code stays local (tree-sitter, no API). Only docs/PDFs go through Anthropic API using your key. ✅

**Known issue (graphify 0.6.8 + Windows):** The hooks installed by `graphify hook install` hardcode `python3`, which doesn't exist on default Windows. The shebang-detection block can't read `graphify.exe` (it's a binary, not a Python script) and falls through to the broken `python3` fallback. Workaround: edit `.git/hooks/post-commit` and `.git/hooks/post-checkout` to point at the uv-managed python:

```
GRAPHIFY_PYTHON="/c/Users/shiva/AppData/Roaming/uv/tools/graphifyy/Scripts/python.exe"
```

If you ever re-run `graphify hook install`, re-apply the patch. Upstream bug to be filed later.

**Known issue (graphify 0.6.8 + .graphifyignore):** the spec'd ignore patterns in the architecture's first draft were not enough — `native-modules/`, `dashboard-next/`, `release/`, `packaging/`, `installer/`, `dist-bundle/`, `electron/`, `workspace-templates/`, `scratch/`, `packages/`, `cloudflare-worker/` all get indexed by default and balloon the graph. Aiden's working `.graphifyignore` excludes all of these in addition to the standard build artifacts. With this list, Aiden builds in ~20s with ~1800 nodes (vs 3+ min and 39k nodes unfiltered).

## The fabrication problem (root cause we're fixing)

**Aiden v3.19.9 today:**
```
User → Planner LLM (call 1, JSON plan)
     → Executor (runs tools)
     → Responder LLM (call 2, never sees raw tool outputs)
     → User
```
Responder is a separate LLM that guesses what happened. It lies:
- "Saved to Desktop/X.md" when no file_write ran
- "Switching to Groq" while still on Ollama
- "Playing song" when 2 tabs opened, nothing played

C20/C21 honesty patches are band-aids. The architecture allows fabrication by design.

**Hermes pattern (Aiden v4.0.0):**
```
User → AidenAgent (ONE LLM, tool-calling loop)
       ├─ LLM picks tool → runs → result returns to LLM context
       ├─ LLM picks next tool OR responds with results in context
       └─ Final response generated by same LLM that saw the tool outputs
     → User
```
One LLM. One context. Tool outputs flow directly into response generation. Lying becomes physically harder because the LLM literally has the tool outputs in its window when generating the response.

## OAuth subscription login — flow detail

Hermes already implements both. We port both patterns.

**Setup wizard prompt:**
```
How do you want Aiden to think?

  [1] Use my Claude Pro/Max subscription      ⭐ Pro feature
  [2] Use my ChatGPT Plus subscription        ⭐ Pro feature
  [3] Use API keys (Groq free, Anthropic, OpenAI, etc.)
  [4] Local only (Ollama, no internet)
```

**Choice 1 — Claude subscription (Pro):**
1. Check `proLicense.ts` — is user Pro? If no, show upgrade message + fall through to choice 3.
2. Check for Claude Code credentials at platform-specific path.
3. If found and refreshable → use directly.
4. If not found → open browser to Anthropic OAuth flow.
5. Token saved to `auth.json`.
6. Hermes-style preflight refresh before every Anthropic call.
7. `anthropic_messages` API mode wired up.

**Choice 2 — ChatGPT subscription (Pro):**
1. Check Pro license.
2. OAuth flow to OpenAI Codex.
3. Token to `auth.json`.
4. `codex_responses` API mode wired up.

**Choice 3 — API keys (Free):**
- Existing flow. Saved to `.env`.

**Choice 4 — Local only (Free):**
- No OAuth, no keys. `ollama_prompt_tools` mode.

**Trust levels:**
- `builtin` (bundled) — always trusted

- `trusted` (verified registries) — permissive policy
- `community` (everything else) — security scan; `--force` overrides cautions, not `dangerous`

**Progressive disclosure:**
```
Level 0: skills_list()        → name + description (~3k tokens for full list)
Level 1: skill_view(name)     → full SKILL.md
Level 2: skill_view(name, path) → reference file (e.g., templates/)
```

**SKILL.md frontmatter (Aiden adopts Hermes spec):**
```yaml
---
name: my-skill
description: Brief description
version: 1.0.0
platforms: [windows, linux, macos]
metadata:
  aiden:
    tags: [trading, automation]
    category: trading
    fallback_for_toolsets: [web]
    requires_toolsets: [terminal]
    config:
      - key: nse.broker
        default: dhan
        prompt: NSE broker
    required_environment_variables:
      - name: DHAN_API_KEY
        prompt: Dhan API key
        help: Get from dhan.co

```

**SkillTeacher Tier 3 (Aiden moat):**
- Watches successful tool-call traces (5+ tools, no errors, user satisfied)
- Proposes new skill via approval prompt: "Save this workflow as a skill?"
- User approves → saved via `skill_manage(action=create)`
- During use, if skill fails 2x, SkillTeacher proposes fix as diff → user approves
- Skill quality scoring: <60% success over 5 runs → flagged
- Pro tier toggle: "Auto mode" enables Hermes-style silent skill evolution

**Total memory cost: ~1,300 tokens per session.**

**Frozen snapshot pattern:**
- Loaded once at session start
- Injected into system prompt as immutable block
- Changes during session: persisted to disk immediately, won't appear in prompt until next session
- Why: preserves prefix cache (Anthropic prompt caching, OpenAI cache hits)
- Tool responses always show live state (not stale)

**Memory tool actions:**
- `add` — new entry (rejected if duplicate)
- `replace` — substring-matched (`old_text` is unique substring)
- `remove` — substring-matched
- No `read` — content already in system prompt

**MemoryGuard (Aiden moat):**
- Wraps every memory tool call
- Verifies the store actually succeeded (not just LLM claim)
- Dual-intent detection: when user says "remember X", confirms LLM both interpreted intent correctly AND fired the tool
- Exposes failures honestly to honesty enforcement layer

**session_search tool:**
- All sessions stored in `sessions.db` (SQLite + FTS5)
- Unlimited capacity (disk-bound)
- Keyword search → Gemini Flash summarization → relevant past conversations
- Pro feature: better embedding-based recall (defer to v4.1.0 polish)

**Memory provider plugins** (8 in Hermes: Honcho, Mem0, Hindsight, Holographic, RetainDB, ByteRover, Supermemory, OpenViking):
- v4.0.0: ship plugin interface, no plugins yet
- v4.1.0: ship at least Honcho + Mem0 plugins

## Cross-platform strategy (Aiden's #1 moat vs Hermes)

| Component | Windows (native) | Linux | macOS |
|---|---|---|---|
| Shell exec | PowerShell | bash | bash |
| Media controls | WinRT (existing) | MPRIS (DBus) | MPMediaPlayer |
| Screen capture | Win32 (existing) | X11/Wayland | ScreenCaptureKit |
| Browser CDP | localhost:9222 | localhost:9222 | localhost:9222 |
| App launch | `start` / Win32 | `xdg-open` | `open` |
| Path constants | `%LOCALAPPDATA%` | `$XDG_DATA_HOME` | `~/Library/Application Support` |
| File watcher | chokidar | chokidar | chokidar |
| Encoding | UTF-8 enforced (avoid cp125x pitfalls) | UTF-8 default | UTF-8 default |

**Approach:** Abstract OS-specific calls behind `platform/{windows,linux,macos}.ts`. Tools call `platform.shellExec()`, not `powershell.execute()`. Same TypeScript core, three OS adapters.

**Hermes is WSL2 on Windows. Aiden is native.** This alone justifies the rewrite for our market.

## Security model (7 layers, full Hermes parity)

| Layer | Aiden v4.0.0 implementation |
|---|---|
| 1. **User authorization** | Stub for v4.1.0 gateway (DM pairing, allowlists) |
| 2. **Dangerous command approval** | `moat/approvalEngine.ts` — manual / smart / off + YOLO + permanent allowlist |
| 3. **Container isolation** | Docker backend (opt-in free, Pro auto for untrusted skills) |
| 4. **Credential filtering** | `.env` separation, MCP env allowlist, file passthrough for OAuth |
| 5. **Context file scanning** | Inject-pattern detection on AGENTS.md, SOUL.md, USER.md |
| 6. **Cross-session isolation** | SQLite per-session keys, path-traversal hardening |
| 7. **Input sanitization** | Working dir allowlist, SSRF protection always-on |

**Dangerous command patterns (port from Hermes `tools/approval.py`):**
- `rm -r`, `rm ... /`, `mkfs`, `dd if=`, fork bombs
- `chmod 777/666/o+w/a+w`, `chown -R root`
- `DROP TABLE`, `DELETE FROM` (no WHERE), `TRUNCATE TABLE`
- `> /etc/`, `> ~/.ssh/`, `tee` to sensitive paths
- `curl ... | sh`, `bash <(curl ...)`, `bash -c`, `python -e`
- `kill -9 -1`, `pkill -9`, `systemctl stop/disable/mask`
- `pkill aiden/gateway` (self-termination prevention)

**Approval flow (CLI):**
```
⚠️  DANGEROUS COMMAND: recursive delete
    rm -rf /tmp/old-project

    [o]nce  |  [s]ession  |  [a]lways  |  [d]eny

    Choice [o/s/a/D]:
```

**Tirith-style content scanner (port from Hermes):**
- Homograph URL detection (international domain attacks)
- Pipe-to-interpreter pattern catch
- Terminal injection attacks
- Auto-install from GitHub releases with SHA-256 verification

**SSRF Protection (always-on, can't be disabled):**
- Block RFC 1918 (10/8, 172.16/12, 192.168/16)
- Block loopback (127/8, ::1)
- Block link-local (169.254/16 — includes cloud metadata at 169.254.169.254)
- Block CGNAT (100.64/10 — Tailscale, WireGuard)
- Block cloud metadata hostnames (`metadata.google.internal`, etc.)

**Approval modes:**
- `manual` (default) — always prompt
- `smart` — auxiliary LLM assesses risk, low-risk auto-approved, high-risk auto-denied, uncertain → prompt
- `off` — YOLO mode (`aiden --yolo`, slash `/yolo`, env `AIDEN_YOLO_MODE=1`)

**Container bypass:** When backend is Docker/Singularity/Modal, dangerous command checks are skipped because container is the security boundary.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Cross-platform port adds bugs | Phase 8-9 manual test matrix on all 3 OSes before merge |
| OAuth flows break (token refresh, expiry) | Port Hermes's `auth.py` patterns directly. Preflight refresh before every call. Fallback to API key on 401. |
| 86 tool migrations introduce regressions | Read-only tools first → write tools → browser/screen last. Sub-test each batch. |
| TypeScript port misses Hermes nuance | Keep Hermes Python repo open as reference. When stuck, read Hermes test files (~3000+ tests). |
| Solo founder timeline slip | Phase-gated. If hit week 16 with major work left, defer cross-platform Linux/macOS to v4.0.1. |
| Existing 1.3k v3.19.9 users break | They don't. v3.19.9 stays live. v4.0.0 ships when ready. |
| OAuth = Pro gating frustrates free users | Free tier still gets full Aiden via API keys. OAuth is convenience, not exclusive. |
| Skills hub trust scanner produces false positives | Start with `--force` override available. Learn from Hermes's curated trust levels. |
| Context window fragmentation across providers | Auxiliary client routes to cheap models. Compression at 50%. Iteration budget at 90 turns. |
| Hermes architecture is Python — TS port introduces concurrency bugs | TypeScript async/await is fine. Use Promise.all for parallel ops. Avoid shared mutable state. |
| Quality drift over 12-16 weeks | Phase-end QA gates before next phase starts. No phase progresses with known regressions. |
